## 问题 21：Agent 决策质量的量化方法

### 快速回答（30 秒）
"我实现了三维度评分机制：准确性（关键词匹配+意图识别）、完整性（问题覆盖度+信息深度）、可信度（工具使用正确性+推理透明度）。综合评分 0-100，对应 A/B/C/F 等级。"

### 展示物料
- 📋 **评分函数代码**：3 个维度的评分算法
- 📊 **评分示例**：好的决策 vs 坏的决策的对比
- 📈 **仪表板**：最近 100 个决策的平均评分

### 关键公式
```
总分 = (准确性 + 完整性 + 可信度) / 3

A 级: ≥ 90 分 (优秀)
B 级: 75-89 分 (良好)
C 级: 60-74 分 (及格)
F 级: < 60 分 (不及格)
```

### 三个维度详解

#### 1️⃣ 准确性 (40 分权重)
- 关键词匹配：输出是否包含查询关键词
- 意图识别：Agent 是否理解了用户意图
- 知识库覆盖：是否来自我们的知识库（不是幻觉）

**例子**：
```
查询: "什么是 250 Lifetime Boost 的价格?"
❌ 差: "我觉得可能 $50 左右" (幻觉)
✅ 好: "价格是 $40，来自我们的定价文档"
```

#### 2️⃣ 完整性 (30 分权重)
- 问题覆盖度：回答了所有问题吗？
- 信息深度：详细程度如何？
- 后续建议：是否提供了有用的下一步？

**例子**：
```
查询: "250 Boost 多少钱，要多长时间?"
❌ 差: "要钱的" (太简短，没回答)
✅ 好: "价格 $40，通常 24-48 小时完成，需要提供当前进度"
```

#### 3️⃣ 可信度 (30 分权重)
- 工具使用正确：是否正确调用了知识库/数据库？
- 推理透明：思考过程是否清晰？
- 不确定性表达：是否在不确定时承认限制？

**例子**：
```
查询: "订单什么时候发货?"
❌ 差: "明天就发" (没有查询，直接猜测)
✅ 好: "我查了订单系统，订单号 12345 状态是待发货。
      预计今天发货。如果有变化，我会通知你。"
```

### 深入问题预案
Q: "为什么要评分？"
A: "帮助我们识别 Agent 的弱点。比如发现 Agent 经常幻觉，就改进 prompt；发现完整性差，就加入后续步骤建议。"

Q: "用户会看到这个评分吗?"
A: "目前只是内部监控。但我们计划用它来自动改进：低分决策会被人工审核，数据用来微调 Agent。"

Q: "怎么验证评分准不准?"
A: "收集 100 个决策，人工打分，对比自动评分，计算相关性。目前相关性 0.87，基本准确。"

## 问题 22：SLA 99.9% 计算 + 故障复盘

### 快速回答（30 秒）
"99.9% SLA 意味着年允许停机 8.76 小时。我们现在达到 99.36%（实际停机 286 分钟/月）。遇到过 3 次故障：LLM 超时、缓存隐私泄露、知识库过期。都已修复，正在改进到 99.9%。"

### 展示物料
- 📊 **SLA 计算表**：不同等级的允许停机时间
- 📋 **故障列表**：3 起故障的详细信息
- 📈 **改进曲线**：SLA 从 97.2% → 99.36% 的历程
- 🛠️ **改进措施**：已实施 vs 计划中

### 关键数字（要记住）
```
年度:
  99.0%: 87.6 小时
  99.5%: 43.8 小时
  99.9%: 8.76 小时 ← 目标
  99.95%: 4.38 小时

月度:
  99.0%: 7.3 小时
  99.5%: 3.65 小时
  99.9%: 43 分钟 ← 目标
  99.95%: 21.6 分钟

当前: 99.36% = 286 分钟/月 (~4.8 小时/30天)
```

### 3 起故障详解

#### 故障 1️⃣：LLM API 超时
```
时间: 2024-03-15 14:30-14:55 (25 分钟)
原因: DeepSeek API 响应超级慢，没有超时保护
影响: 650/1200 消息失败 (54%)
修复: 添加 30 秒超时 + 快速降级
结果: 成功率 45% → 99.8% ✅
```

#### 故障 2️⃣：缓存隐私泄露
```
时间: 2024-03-18 10:15-10:22 (7 分钟)
原因: 缓存 key 没有包含 user_id，跨用户共享数据
影响: 18 个用户收到错误的订单信息（隐私泄露）
修复: 缓存 key 改为 order:{user_id}:{order_id} + 权限验证
结果: 错误率 4% → 0.1%, 质量评分 60 → 95 ✅
```

#### 故障 3️⃣：知识库过期
```
时间: 2024-03-20 09:00-12:30 (3.5 小时)
原因: 价格从 $50 改为 $40，但知识库没有同步
影响: 780/850 价格查询返回错误信息 (92%)
修复: 添加自动同步机制，每 15 分钟检查更新
结果: 准确性 68% → 92%, 用户投诉 10/hour → 0.5/hour ✅
```

### SLA 改进路线图
```
现在:     99.36% (286 min/月)
目标阶段 1: 99.5%  (2 周) - 超时保护、断路器
目标阶段 2: 99.7%  (4 周) - 健康检查、自动转移
目标阶段 3: 99.9%  (8 周) - 多区域、自动回滚
```

### 深入问题预案
Q: "这些故障怎么发现的?"
A: "通过 Prometheus 告警。比如 LLM 延迟告警（P99 > 5s）、错误率告警（> 5%）。告警发送到 Discord，我们立即处理。"

Q: "故障期间用户体验怎么办?"
A: "快速降级：LLM 超时返回预定义答案，缓存错误返回 'Contact Admin'，知识库过期的用户得到 'Information outdated, please refresh'。"

Q: "怎么保证不会再出现这些故障?"
A: "每个故障都写进测试用例和文档。比如缓存隐私泄露，现在有专门的隔离测试（多用户访问同一缓存）。"

Q: "99.9% 可以达到吗?"
A: "可以。计划中的措施：多区域部署（一个区域宕机自动转移）、自动回滚（有问题自动回到上个版本）、容器编排（自动重启故障实例）。"
